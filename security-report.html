Critical Findings

SEC-01   MFA Bypass with Hardcoded OTP "000000"
OWASP A07:2021 — Identification and Authentication Failures
Critical Accepted Risk (POC)
AuthService.java:68–78  |  application-aws.yml

A hardcoded mfaBypassEmails list allows specific email addresses to skip MFA entirely and receive a JWT token directly on login. The code stores a fixed OTP of "000000" for these accounts. This bypass was introduced because email delivery via Amazon SES was not available for the admin account during POC development.

// POC bypass — skips MFA for configured emails if (mfaBypassEmails.contains(request.getEmail())) { otpService.storeOtp(request.getEmail(), "000000"); return AuthResponse.builder() .accessToken(jwtService.generateAccessToken(user)) .build(); }
Risk Accepted — POC Only: This bypass is intentionally retained for this POC assignment due to email delivery constraints. The bypass email is now configured via the ${MFA_BYPASS_EMAILS} environment variable rather than hardcoded in source. This mechanism must be removed before any production deployment and replaced with a properly configured SES-verified email or TOTP-based admin authentication.
SEC-02   Refresh Cookie Secure Flag Hardcoded to false
OWASP A02:2021 — Cryptographic Failures
Critical Fixed
AuthController.java:121

The refresh token cookie was created with setSecure(false), meaning it could be transmitted over plain HTTP connections. A comment stated it should be set to true for AWS, but this was never enforced in code — any misconfigured deployment would silently transmit refresh tokens in the clear, allowing them to be intercepted by network attackers.

cookie.setSecure(false); // set true in aws profile via nginx/API Gateway — never enforced!
Fix Applied: Replaced the hardcoded false with a configurable property app.cookie.secure (defaults to true). Set app.cookie.secure: false in application-local.yml for local H2 dev only. Production (aws profile) inherits the true default.
SEC-03   Refresh Token Type Not Validated (Token Confusion Attack)
OWASP A07:2021 — Identification and Authentication Failures
Critical Fixed
AuthService.java:110–118  |  JwtService.java

The refreshToken() method validated that the supplied token was a valid, unexpired JWT signed by the correct key — but never checked whether it was actually a refresh token. Both access tokens and refresh tokens are signed with the same RSA private key. An attacker in possession of a stolen access token (e.g., from a compromised log, browser cache, or XSS) could submit it to the /api/auth/refresh endpoint and obtain a fresh access token, extending their session indefinitely beyond the 15-minute access token TTL.

// VULNERABLE — validates signature + expiry, but not token type public AuthResponse refreshToken(String refreshToken) { String email = jwtService.extractUsername(refreshToken); User user = ...findByEmail(email)... if (!jwtService.isTokenValid(refreshToken, user)) { // no type check! throw new InvalidOtpException("Invalid or expired refresh token"); } ... }
Fix Applied: Added isRefreshToken(String token) to JwtService which reads the "type" claim and asserts it equals "refresh". The refreshToken() method in AuthService now validates the type claim as its first check, rejecting any access token submitted to the refresh endpoint.

High Findings

SEC-04   X-Forwarded-For Spoofing Bypasses Rate Limiting
OWASP A05:2021 — Security Misconfiguration
High
RateLimitFilter.java:49–52  |  AuthController.java:152–154

Both the rate limiter and the audit log trust the client-supplied X-Forwarded-For header directly, taking the first IP in the comma-separated chain. An attacker can set X-Forwarded-For: 1.2.3.4 in every request to rotate identities and bypass the 100 req/min limit entirely.

return xForwardedFor.split(",")[0].trim(); // attacker-controlled first value
Recommended Fix: Trust only the last IP added by your load balancer (the rightmost untrusted-network entry). Use request.getRemoteAddr() when behind a single trusted proxy, or parse only the last value: xForwardedFor.split(",")[last].trim().
SEC-05   OTP Brute Force Not Prevented
OWASP A07:2021 — Identification and Authentication Failures
High
AuthController.java:74  |  InMemoryOtpService.java

The POST /api/auth/mfa/verify endpoint has no per-user attempt counter or lockout. A 6-digit OTP has only 1,000,000 combinations. Combined with the IP spoofing issue (SEC-04), an attacker can brute force a valid OTP within hours at scale.

Recommended Fix: Track failed OTP attempts per email in the OTP store. Invalidate the OTP and lock the account for N minutes after 5 consecutive failures. Re-issue a new OTP only after the lockout expires.
SEC-06   User Enumeration via Registration Error Messages
OWASP A01:2021 — Broken Access Control
High
AuthService.java:40, 43

Registration errors reveal whether a specific email or username already exists in the system. An attacker can enumerate valid accounts by posting to /api/auth/register and reading the distinct error messages.

throw new DuplicateResourceException("Email already registered: " + request.getEmail()); throw new DuplicateResourceException("Username already taken: " + request.getUsername());
Recommended Fix: Return a generic message regardless of conflict type: "Registration failed. Please check your details and try again."

Medium Findings

SEC-07   Swagger UI Publicly Accessible in All Environments
OWASP A05:2021 — Security Misconfiguration
Medium
SecurityConfig.java:53–58

/swagger-ui/** and /api-docs/** are marked permitAll(), exposing the full API contract to unauthenticated callers in production — including all endpoint paths, request/response schemas, and authentication flows.

Recommended Fix: Add @Profile("local") to the OpenAPI configuration bean, or require ADMIN role to access Swagger in production.
SEC-08   Refresh Token Cookie TTL Miscalculated (12h Instead of 7 Days)
OWASP A07:2021 — Identification and Authentication Failures
Medium
AuthController.java:123

The cookie max-age calculation uses the access token expiration (900,000 ms = 15 min) multiplied by 48, giving 43,200 seconds = 12 hours. The comment states "7-day TTL" but the actual TTL is 12 hours, causing silent user session loss and forcing unnecessary re-logins.

cookie.setMaxAge((int) (jwtService.getExpiration() / 1000 * 48)); // 900s × 48 = 12h, NOT 7 days
Recommended Fix: Expose getRefreshExpiration() from JwtService and use it: cookie.setMaxAge((int) (jwtService.getRefreshExpiration() / 1000)).
SEC-09   Admin Controller Bypasses Service Layer Audit Logging
OWASP A04:2021 — Insecure Design
Medium
AdminController.java:49–52

AdminController directly injects LogRepository, UserRepository, TransactionRepository, and AppSettingRepository. Direct repository calls bypass any business logic and audit trail that belongs in the service layer. Admin actions (viewing logs, reading all users) go unrecorded.

Recommended Fix: Move all data access to service methods. Controllers should never hold repository references directly.
SEC-10   SameSite Attribute Not Set on Refresh Cookie
OWASP A05:2021 — Security Misconfiguration
Medium
AuthController.java:118–125

The refresh cookie is set without a SameSite attribute. While modern browsers default to Lax, explicitly setting SameSite=Strict prevents the cookie from being sent in any cross-site context.

Recommended Fix: Switch to Spring's ResponseCookie builder which supports .sameSite("Strict"):
ResponseCookie.from("refresh_token", token).httpOnly(true).secure(true).sameSite("Strict")...

SEC-11   External API Key Defaults to "demo"
OWASP A05:2021 — Security Misconfiguration
Low
AlphaVantageClient.java:19  |  application.yml:70

The Alpha Vantage API key falls back to the string "demo" if not configured. The demo key is shared across all demo users and is heavily rate-limited, causing silent degradation in production if the env var is not set.

Recommended Fix: Remove the default value to force a startup failure if the key is absent. Add a @PostConstruct validation or use @NotBlank on the field.
SEC-12   JWT Public Key Logged at INFO Level
OWASP A09:2021 — Security Logging and Monitoring Failures
Low
JwtService.java:51

The auto-generated RSA public key is logged at INFO level. While the public key is by definition not secret, emitting it into application logs is unexpected and may be flagged by security scanners or log monitoring tools.

log.info("RSA keys generated. Public key (base64): {}", ...);
Recommended Fix: Downgrade to log.debug(...).
