🎯 Q&A Preparation — 15 Likely Topics
ℹ️

Panel questions will come from three angles: why did you choose X over Y (architecture decisions), what breaks if Z fails (failure modes), and how would you evolve this (scale / production). Prepare answers in that shape.

Q1 Why JWT RS256 instead of the more common HS256? Security
Short answer: Asymmetric keys — only this service holds the private key; any other service can verify tokens with just the public key.

HS256 uses a shared secret. Every service that needs to verify tokens must have that secret — which means the secret leaks as you add services. RS256 uses a key pair: the private key signs tokens (only this backend), the public key verifies them. You can distribute the public key freely. This is microservices-ready by design, and fits the CQRS/event-driven future-state where separate read services may need to verify caller identity without having the signing key.
Q2 What happens if Redis goes down? SecurityOps
Three consequences, in order of severity:
  • MFA breaks — OTP codes are stored in Redis with a 5-minute TTL. If Redis is down, verifying the OTP fails. Users mid-login cannot complete authentication.
  • Token blacklist lost — Logged-out tokens are no longer blacklisted. Any attacker holding a stolen JWT can use it freely until its natural 15-minute expiry. This is a security regression.
  • Rate cache gone — Every rate request hits the external API directly. The circuit breaker may open if the external API rate-limits us. Exchange rates still serve (from API), but latency spikes.
Production mitigation: Redis primary + replica with automatic ElastiCache failover (under 60 seconds). The replica holds a copy of all data, so OTPs and the blacklist survive a primary failure.
Q3 Why cache-aside instead of write-through caching for exchange rates? Architecture
Write-through requires you to own the write path. In write-through, every time you write data, you simultaneously update the cache. But exchange rates come from an external API — you do not control when they change. There is no write event you can intercept to update the cache. Cache-aside is the correct pattern when data originates externally: you populate the cache on demand, and let TTL expiry handle staleness. If you also have admin-set manual rates, cache eviction on those writes is handled by @CacheEvict — a targeted invalidation, not a full write-through.
Q4 How does the approval threshold work across different currencies? Design
The threshold is defined in USD and all comparisons happen in USD equivalent. When a user converts GBP → JPY, the service first calls getRate("GBP", "USD") to get the GBP-to-USD rate, multiplies by the GBP amount to get the USD equivalent, then compares that against the threshold. Only if fromCurrency is already USD is this extra call skipped. This ensures currency-agnostic fairness — a £80 conversion and a $100 conversion are treated identically regardless of the currency pair the user chose.
Q5 Why BCrypt cost factor 12 for passwords? Security
BCrypt's cost factor controls how many iterations the hashing algorithm runs. Cost 12 produces a hash in approximately 250ms on modern hardware. This is deliberate: it makes offline brute-force attacks expensive. An attacker with a GPU farm cracking a leaked hash database can only test ~4 guesses per second per GPU at cost 12, compared to millions of guesses per second for unsalted SHA-256. OWASP recommends cost 10–12. Cost 12 is slightly more conservative and still within acceptable login latency. As hardware gets faster, the cost can be increased — BCrypt's design allows this without invalidating existing hashes (existing passwords re-hash at login with the new cost).
Q6 How would you scale this system to handle 10x traffic? Scale
The application is stateless by design — JWT tokens, OTPs, and the rate cache are all in Redis, not on the instance. Scaling horizontally means adding more ECS Fargate tasks behind an Application Load Balancer. Each task reads from the same Redis and writes to the same RDS — no sticky sessions needed.

Bottlenecks at 10x:
  • RDS connections — HikariCP pool per task (5 connections). At 20 tasks that's 100 connections — within RDS limits for db.t3.medium. Beyond that, add PgBouncer as a connection pooler.
  • Redis memory — The rate cache is small (one key per currency pair). The OTP and blacklist data grows with user count. Upgrade to cache.r6g.large (13 GB) for headroom.
  • External API rate limits — Add the background scheduler to pre-warm popular pairs, so concurrent traffic doesn't all hit the external API on cache miss simultaneously.
Q7 Explain the circuit breaker configuration and what triggers it. Architecture
Resilience4j is configured with: 50% failure threshold over a sliding window of 10 calls, 30-second open wait, 3 permitted calls in half-open state. When the primary exchange rate API fails 5 out of 10 consecutive calls, the circuit opens. All subsequent calls immediately throw an ExternalApiException without touching the network — fast-fail. After 30 seconds, the circuit moves to half-open and allows 3 test calls. If those succeed, it closes; if they fail, it opens again.

The fallback chain is: primary API → Alpha Vantage (secondary). The circuit breaker wraps only the primary call. If the primary is down, the fallback fires directly. If both are down, ExternalApiException propagates to the global handler which returns 503 with a user-friendly message — no stack trace leaked.
Q8 How do you prevent a logged-out JWT from being reused? Security
On logout, the access token is added to a Redis blacklist with a TTL equal to its remaining lifetime. On every authenticated request, JwtAuthFilter checks TokenBlacklistService.isBlacklisted(token) before doing anything else. If the token is blacklisted, the filter passes through without setting the security context — the request proceeds as unauthenticated and Spring Security returns 401.

The TTL on the blacklist entry matches the token's natural expiry, so Redis auto-cleans entries. There is no growing blacklist — it self-prunes. The attack window if Redis is down is at most 15 minutes (access token TTL), after which the token expires naturally.
Q9 Why Elastic Beanstalk for this POC instead of ECS or Lambda? Ops
EB vs ECS: EB abstracts EC2 management — no Dockerfile, no task definitions, no ECS cluster configuration. For a POC where the goal is demonstrating application functionality, EB lets you deploy a JAR in minutes. ECS Fargate is the right production target (as documented) but adds 2–3 hours of setup for container registry, task definitions, service configuration, and ALB wiring that adds no POC value.

EB vs Lambda: Spring Boot's startup time (2–4 seconds) makes Lambda cold starts painful for a stateful, session-oriented app. Lambda also has a 15-minute execution limit and no persistent connections — HikariCP and Redis pooling both break in a Lambda model. Spring Boot's design is aligned with long-running processes, not function invocations.
Q10 Why is the refresh token in an HttpOnly cookie instead of the response body? Security
An HttpOnly cookie is inaccessible to JavaScript — document.cookie cannot read it, so an XSS attack that injects malicious JavaScript into the page cannot steal the refresh token. If the refresh token were in the response body and stored in localStorage or even a regular cookie, any successful XSS would allow an attacker to silently rotate tokens indefinitely, maintaining persistent access long after the user has logged out.

The access token (15-minute lifetime, in the response body) has limited XSS exposure — even if stolen, it expires quickly. The refresh token (7-day lifetime) is the higher-value target, so it gets the stronger protection. The cookie is also scoped to Path=/api/auth — it is only sent on auth endpoints, not on every API call, minimising its surface area further.
Q11 What is the biggest risk of this POC architecture in production? Ops
Single points of failure everywhere:
  • Single EC2 instance — hardware failure or AZ outage takes the entire app offline. No standby, no failover.
  • Single Redis node — as discussed: MFA breaks, blacklist lost, cache cold.
  • Single-AZ RDS — if the AZ goes down, transaction data is unreachable. The financial records of a real system cannot be in a single-AZ database.
  • Plaintext credentials in EB env vars — if someone gains access to the EB console, they see the DB password, JWT private key, and SES credentials in plaintext.
The documented production path addresses all four with ECS Fargate Multi-AZ, Redis primary+replica, RDS Multi-AZ, and AWS Secrets Manager respectively.
Q12 Why is the exchange rate refresh demand-driven and not on a schedule? Design
Five reasons the background scheduler was excluded from the POC:
  • No cold-start problem at POC scale — 1–2 demo users never hit the same expired pair concurrently.
  • API quota cost — Free-tier APIs have call limits. A scheduler refreshing 10 pairs every 4 minutes burns ~3,600 calls/day whether the app is used or not.
  • No usage data for pair selection — The scheduler needs to know which pairs are worth pre-warming. That comes from historical transaction volume, which doesn't exist for a new system.
  • Startup thundering herd complexity — Correct implementation requires staggered delays on startup to avoid hammering the external API simultaneously for all pairs.
  • 5-minute TTL satisfies the POC SLA — "Real-time" in a demo context means not hours old.
In production, the scheduler would read top-20 pairs from transaction history and pre-warm them before TTL expires.
Q13 How would you add event-driven architecture or CQRS to this system? ArchitectureScale
The current system is a synchronous monolith — all operations are in-request. The natural evolution:
  • Event sourcing for transactions — Instead of mutating status in place, publish events: TransactionCreated, TransactionApproved, TransactionRejected to SQS or EventBridge. The current Log entity is a primitive version of this already.
  • CQRS for the admin dashboard — The dashboard currently runs 5 aggregation queries on the transactional DB. A separate read model (materialized view or a DynamoDB projection updated by events) serves the admin dashboard without competing with write traffic.
  • Async email notifications — Currently SES is called synchronously in the transaction flow. Moving to SQS + a Lambda consumer decouples email delivery failure from transaction success.
The application boundaries are already clean — controllers delegate to services, services to repositories — so event publishing could be added at the service layer without restructuring.
Q14 How is RBAC enforced — what stops a regular user from calling admin APIs? Security
Two layers, both active:

1. URL-level: SecurityConfig declares .requestMatchers("/api/admin/**").hasRole("ADMIN"). Spring Security evaluates this before the request reaches any controller. A USER-role JWT hitting /api/admin/transactions gets a 403 at the filter chain level — no controller code runs.

2. Method-level: @PreAuthorize("hasRole('ADMIN')") on every admin controller method and on admin-only rate management endpoints. Even if someone bypassed the URL matcher, method security provides a second check. This is defence in depth — two independent enforcement points.

The role is embedded in the JWT at issuance time (the User.getAuthorities() method returns ROLE_USER or ROLE_ADMIN) and is verified on every request by JwtAuthFilter.
Q15 What would you do differently if you started this project again? Architecture
Three things:
  • Add Micrometer custom metrics from day one. The current /actuator/metrics only exposes JVM defaults. Business counters for transactions created, MFA failures, circuit breaker opens, and threshold breaches would have made the observability story much stronger without significant extra effort.
  • Use a proper JSON logging encoder from the start. The initial AWS logback pattern was a string that looked like JSON but was not valid JSON — special characters in log messages would have broken CloudWatch Insights parsing. Logback's built-in JsonEncoder was a one-line fix that should have been the first choice.
  • Decouple email sending from the transaction flow with SQS. Currently, if SES is slow or unavailable, the entire conversion request hangs waiting for the email call to return. Making the email step asynchronous via a queue would make the conversion path faster and more resilient to email service issues.
