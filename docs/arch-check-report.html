Rule 1 — Controllers must not contain business logic
VIOLATION Business logic in AuthController.login()
File
AuthController.java:63–73
Issue
The login() method queries UserRepository directly, calls authService.generateRefreshToken(), and builds and sets the refresh cookie — all inside the controller. The if (auth != null) branch that drives the MFA-required vs MFA-bypass behaviour is business logic that belongs in AuthService.
Fix
Move refresh-token generation and cookie-building into AuthService.login() or a new AuthService.completeLogin() method; return a wrapper response DTO that the controller can blindly pass back.
VIOLATION Data transformation logic in AdminController
File
AdminController.java:170–179
Issue
The private toLogResponse(Log log) mapper lives inside the controller. Controllers must not own mapping logic — it belongs in a service or dedicated mapper class.
Fix
Move toLogResponse() into LogService (or a LogMapper component) and have the service return Page<LogResponse> directly, eliminating the mapper from the controller entirely.
Rule 2 — Repositories must not be injected into controllers directly
VIOLATION UserRepository injected into AuthController
File
AuthController.java:45 (used at lines 67, 85)
Issue
UserRepository is injected into the controller and called in login() and verifyMfa() to fetch the User entity for refresh-token generation. This is not a simple dashboard aggregation — it is business-logic-adjacent and must go through AuthService.
Fix
Add an AuthService method that accepts email and returns the refresh token; remove the UserRepository field from AuthController.
VIOLATION AppSettingRepository injected into AdminController (beyond dashboard)
File
AdminController.java:50 (used at lines 132–137, 146)
Issue
AppSettingRepository is injected and used directly in getApprovalThreshold() and updateApprovalThreshold(). AppSettingService already owns this domain — the controller bypasses the service for the read path. The documented exception only covers simple read-only dashboard aggregations.
Fix
Extend AppSettingService to return a full ApprovalThresholdResponse and remove the AppSettingRepository injection from the controller.
VIOLATION LogRepository injected into AdminController (non-dashboard)
File
AdminController.java:51 (used at lines 158–168)
Issue
LogRepository is injected and used in getLogs() and getLogsByType() — these are full data-access endpoints, not dashboard aggregations. The documented exception does not cover them.
Fix
Add getLogs(Pageable) and getLogsByType(String, Pageable) to LogService (returning Page<LogResponse>) and delegate from the controller.
Rule 3 — @Valid required on all @RequestBody parameters
VIOLATION Missing @Valid on Map request body — UserController
File
UserController.java:35
Issue
@RequestBody Map<String, String> body has no @Valid annotation. While a raw Map carries no Bean Validation constraints, the rule requires @Valid on every @RequestBody for consistency and future-proofing.
Fix
Replace the Map body with a typed request DTO carrying @Valid, or at minimum add @Valid to the existing parameter.
VIOLATION Missing @Valid on Map request body — AdminController.reject()
File
AdminController.java:122
Issue
@RequestBody(required = false) Map<String, String> body — same issue as above.
Fix
Introduce a RejectRequest DTO with a nullable reason field and annotate with @Valid @RequestBody(required = false).
Rule 4 — Write operations in services must be @Transactional
VIOLATION Unguarded write inside @Cacheable method — ExchangeRateService.getRate()
File
ExchangeRateService.java:36–59
Issue
getRate() has no @Transactional but calls saveOrUpdateRate() (line 57) on a cache miss, which calls rateRepository.save(). Each save inside the ifPresentOrElse lambda runs in its own auto-committed repository-level transaction rather than a single service-level unit. @Transactional and @Cacheable on the same method also creates a proxy-ordering conflict.
Fix
Extract the fetch-and-persist logic into a separate @Transactional private fetchAndPersistRate() method called from getRate() after the cache miss, keeping @Cacheable and @Transactional on separate methods.
VIOLATION Async write method missing @Transactional — LogService.log()
File
LogService.java:19
Issue
log() is annotated @Async and calls logRepository.save() but has no @Transactional. Async methods spawn a new thread with no inherited transaction context. If the save fails, there is no rollback and the failure is silently swallowed.
Fix
Add @Transactional(propagation = Propagation.REQUIRES_NEW) to log() so each async log entry runs in its own committed transaction with proper error surfacing.
